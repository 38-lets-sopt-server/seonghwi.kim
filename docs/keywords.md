# 1주차 키워드 과제

## 1. `final` / `static` / `static final` 의 차이?

### `final`
`final`은 한 번 정한 값을 다시 바꿀 수 없게 합니다. 
변수에 붙이면 다시 대입이 불가능하고, 메서드에 붙이면 오버라이딩할 수 없고, 클래스에 붙이면 상속할 수 없습니다.

예를 들어
```java
final PostRepository postRepository = new PostRepository();
```
`postRepository`가 다른 객체를 다시 참조하도록 바꿀 수 없습니다.  
다만 `final`은 참조를 고정하는 것이지, *객체 내부 상태까지 항상 변경 불가능하다는 뜻은 아닙니다.*

---

### `static`
`static`은 객체가 아니라 클래스에 소속되는 요소를 만들 때 사용합니다.
따라서 객체를 생성하지 않아도 클래스 이름으로 바로 접근할 수 있습니다.

예를 들어 
```java
ApiResponse.success();
```
같은 메서드는 객체를 생성하지 않고도 사용할 수 있는데, 이런 메서드가 `static` 메서드입니다.

---

### `static final`
`static final`은 클래스에 하나만 존재하면서, 값도 바꿀 수 없는 상수를 의미합니다.  
즉, 여러 객체가 함께 공유하면서도 절대 변하지 않는 값을 만들 때 사용합니다.

예를 들어 게시글 제목의 최대 길이를
```java
private static final int MAX_TITLE_LENGTH = 50;
```
으로 두면, 이 값은 모든 객체가 같이 사용하고 수정도 불가능합니다.

---

### 정리

- `final` : 값을 다시 바꿀 수 없게 함
- `static` : 객체가 아니라 클래스에 소속됨
- `static final` : 클래스가 공유하는 변경 불가능한 상수

---

## 2. Java의 Generic 타입이란 무엇인가요?

Generic은 클래스나 메서드가 사용할 자료형을 미리 고정하지 않고, 사용할 때 타입을 정할 수 있게 해주는 문법입니다.

예를 들어
```java
List<PostResponse>
```
에서 `<PostResponse>` 부분은 Generic 타입을 지정한 부분입니다.
이 뜻은 이 리스트에는 `PostResponse` 타입만 넣겠다는 의미입니다.

---

Generic을 사용하는 이유는 다음과 같습니다.

### 1) 타입 안정성
리스트나 응답 객체 안에 아무 타입이나 들어가는 것을 막을 수 있습니다.  
즉, 잘못된 타입을 넣는 실수를 줄일 수 있습니다.

### 2) 형변환을 줄일 수 있음
Generic이 없으면 데이터를 꺼낼 때마다 형변환을 해야 하는 경우가 많습니다.  
하지만 Generic을 사용하면 꺼낼 때 이미 타입이 정해져 있어서 불필요한 형변환 코드를 줄일 수 있습니다.

### 3) 재사용성이 좋아짐
하나의 클래스 구조를 여러 타입에 대해 재사용할 수 있습니다.

예를 들어 이번 과제에서 만든
```java
ApiResponse<T>
```
는 Generic을 사용한 클래스입니다.  
그래서 상황에 따라 다음처럼 다양하게 사용할 수 있습니다.

- `ApiResponse<Long>` : 게시글 생성 후 생성된 ID를 담는 경우
- `ApiResponse<PostResponse>` : 게시글 단건 조회 결과를 담는 경우
- `ApiResponse<List<PostResponse>>` : 게시글 전체 조회 결과를 담는 경우
- `ApiResponse<Void>` : 수정, 삭제처럼 반환할 데이터가 없는 경우

즉, Generic은 **하나의 구조를 여러 타입에 맞게 유연하게 사용할 수 있도록 해주는 문법**입니다.

---

## 3. Controller / Service / Repository / Domain 은 각각 어떤 역할을 하나요?

이번 과제에서 사용한 구조를 기준으로 보면 각 계층은 역할이 나뉘어 있습니다.

### Controller
Controller는 클라이언트(Main)의 요청을 가장 먼저 받는 계층입니다.  
사용자가 어떤 기능을 요청했는지 받고, 그 요청을 Service에 전달한 뒤, 결과를 응답 형태로 돌려주는 역할을 합니다.

이번 과제에서는 `PostController`가 이 역할을 담당했습니다.

---

### Service
Service는 비즈니스 로직을 처리하는 계층입니다.  
즉 단순히 데이터를 저장하거나 꺼내는 것이 아니라, 게시글 생성, 조회, 수정, 삭제 같은 실제 처리 흐름을 담당합니다.

이번 과제에서는 `PostService`가
- 게시글 생성
- 전체 조회
- 단건 조회
- 수정
- 삭제

를 처리했고, 게시글이 없을 때 예외를 발생시켜 잘못된 요청을 적절하게 처리하도록 구현했습니다.

---

### Repository
Repository는 데이터를 저장, 조회, 삭제하는 계층입니다.  
즉, 데이터에 접근하여 저장, 조회, 삭제 등의 작업을 수행합니다. (*쉽게 말하면 저장소 역할을 함!!*)

이번 과제에서는 `PostRepository`가 리스트를 이용해서 게시글을 저장하고,
- `save()`
- `findAll()`
- `findById()`
- `deleteById()`

같은 메서드로 데이터를 관리했습니다.

---

### Domain
Domain은 서비스에서 다루는 핵심 데이터를 표현하는 객체입니다.  
이번 과제에서는 `Post` 클래스가 Domain에 해당합니다.

`Post` 안에는 게시글의 주요 데이터인

- id
- title
- content
- author
- createdAt

같은 데이터가 들어 있습니다.  
또한 `update()` 메서드를 통해 객체 자신의 상태를 변경하는 기능도 포함하고 있습니다.

---

### 정리

- `Controller` : 요청을 받고 응답을 반환하는 역할
- `Service` : 비즈니스 로직을 처리하는 역할
- `Repository` : 데이터를 저장, 조회, 삭제하는 역할
- `Domain` : 핵심 데이터 구조를 표현하는 역할
